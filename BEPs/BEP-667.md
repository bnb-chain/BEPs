## 1. Summary

This BEP introduces a **Vote Interval (N)** parameter to the Fast Finality mechanism ([BEP-126](./BEP126.md)), relaxing the consensus constraint from per-block voting to per-N-block voting. This breaks the fundamental `BlockInterval > 2 × RTT` physical limit, maximizing the time window available for transaction packing. **The initial value of N is 2.**

## 2. Abstract

Under the current Fast Finality design ([BEP-126](./BEP126.md)), validators must complete a full vote-propagation cycle within each block interval: receive the block, cast a BLS vote, and propagate it to ⅔+ validators before the next block is proposed. This imposes a hard physical constraint:

```
BlockInterval > 2 × RTT + T_import
```

where `RTT` is the network round-trip time between validators (~120ms) and `T_import` is the block import/verification time.

This BEP decouples the voting frequency from the block production frequency by introducing a configurable Vote Interval `N`. Validators only vote on every Nth block, and attestations are only aggregated and included in the header of every Nth block. The constraint is relaxed to:

```
N × BlockInterval > 2 × RTT + T_import
```

With `N = 2`, the block interval can be reduced to 250ms — well below the `2 × RTT ≈ 240ms` barrier — while the effective vote cycle of 500ms still comfortably satisfies the propagation requirement. CPU utilization for transaction packing improves from ~46.7% to ~88%.

## 3. Motivation

### 3.1 The Physical Limit of Per-Block Voting

The current block interval of 0.45 seconds on BSC ([BEP-619](./BEP-619.md)) is approaching the physical limit imposed by Fast Finality's per-block voting requirement.

[BEP-126](./BEP126.md) assumes that "1 block time is totally enough for validators to receive a block, vote for the block and propagate the vote to other validators." This assumption requires:

```
BlockInterval > 2 × RTT + T_import
```

With inter-validator RTT of ~120ms, the minimum block interval for per-block voting is approximately `2 × 120 + 30 = 270ms`. The current 450ms interval satisfies this but leaves substantial room consumed by vote propagation rather than productive work.

### 3.2 CPU Utilization Under Current Constraint

The timeline for a single block cycle under per-block voting (`N = 1`):

```
|<------------ BlockInterval (450ms) ------------>|
|  T_const  |           T_fill (~210ms)           |
|  (~240ms) |                                     |
```

Where `T_const` includes block import, seal, and the vote-propagation wait overhead (~240ms), and `T_fill` is the time available for transaction packing. This yields:

```
CPU Utilization ≈ T_fill / BlockInterval = 210 / 450 ≈ 46.7%
```

More than half of each block interval is consumed by consensus overhead rather than transaction processing. As block intervals decrease further, this ratio worsens — the constant `2 × RTT` cost takes an ever-larger share.

By voting every N blocks, the `2 × RTT` cost is amortized across N intervals, dramatically increasing the fraction of time available for packing.

## 4. Specification

### 4.1 Vote Interval Parameter

Introduce a new consensus parameter:

```
VoteInterval = N    (initial value: 2)
```

where `N ≥ 1` is a positive integer. When `N = 1`, the behavior is identical to the current per-block voting model.

A block is defined as a **voting block** if:

```
blockNumber % N == 0
```

Only voting blocks participate in the Fast Finality vote-aggregate-justify cycle. Non-voting blocks are produced and validated normally but do not carry vote attestations in their headers and do not trigger voting.

### 4.2 Modified Validator Vote Rules

The vote rules from [BEP-126](./BEP126.md) Section 4.1.1 are adapted to operate on voting blocks:

1. Validators only cast votes for **voting blocks** (`blockNumber % N == 0`).
2. The source `s` and target `t` of a vote message `<v, s, t, h(s), h(t)>` refer to voting blocks only, where `h()` denotes the voting block's sequence number (i.e., `blockNumber / N`).
3. All existing vote rules apply to the voting block sequence:
   - **Rule 1**: A validator must not publish two distinct votes for the same voting block height.
   - **Rule 2**: A validator must not vote within the span of its other votes.
   - **Rule 3**: Validators always vote for their canonical chain's latest voting block.

### 4.3 Modified Aggregate Vote Rules

The aggregate vote rules from [BEP-126](./BEP126.md) Section 4.1.2 are modified as follows:

1. When a validator proposes a **voting block** (i.e., `blockNumber % N == 0`), if the previous voting block (at `blockNumber - N`) has gathered ⅔+ votes from valid validators, the proposer aggregates these votes as an attestation for that previous voting block.
2. The attestation is set into the extra field of the voting block's header.
3. Non-voting blocks (`blockNumber % N != 0`) MUST NOT carry vote attestations in their headers.

### 4.4 Modified Finality Rules

The finality rules from [BEP-126](./BEP126.md) Section 4.1.3 are modified to operate on voting blocks:

1. A voting block is called **justified** if:
   - (1) it is the root, or
   - (2) there exists an attestation for this block in the next voting block's header.
2. A voting block is called **finalized** if:
   - (1) it is the root, or
   - (2) it is justified and the next voting block after it is also justified.
3. When a voting block is finalized, **all blocks between it and the previous finalized voting block are also considered finalized**.

### 4.5 Epoch Alignment

The Epoch length MUST be a multiple of `VoteInterval (N)` to ensure the last block of each epoch is a voting block. This guarantees that validator set transitions occur at voting block boundaries, preventing edge cases where a new validator set takes effect mid-voting-cycle.

With `N = 2`, the Epoch length must be even. The current Epoch of 1000 blocks satisfies this requirement.

### 4.6 Parameter Changes

| Parameter | Current (N=1, 0.45s) | With VoteInterval N=2 | Notes |
| --------- | --------------------- | --------------------- | ----- |
| VoteInterval (N) | 1 (implicit) | **2** | New parameter |
| BlockInterval | 450ms | 250ms | Enabled by relaxed vote constraint |
| Effective Vote Cycle | 450ms | 500ms (2 × 250ms) | Must exceed 2 × RTT + T_import |
| GasLimit | 55M | TBD | Proportional adjustment needed |
| Epoch | 1000 | TBD | Must be a multiple of N |
| TurnLength | 16 | 16 | Unchanged; 8 voting blocks per turn |
| KAncestorGenerationDepth | 3 | TBD | Applies to voting-block generations |
| receiveRateLimitPerSecond | 68 | TBD | May decrease as voting frequency halves |

## 5. Performance Analysis

### 5.1 Traditional Mode (N=1, BlockInterval=450ms)

```
Constraint:     BlockInterval(450ms) > 2 × RTT(240ms) + T_import(30ms) ✓
T_fill         = 450ms - T_const(~240ms) ≈ 210ms
CPU Utilization = 210 / 450 ≈ 46.7%
```

The validator spends more than half of each interval on consensus overhead rather than packing transactions.

### 5.2 Optimized Mode (N=2, BlockInterval=250ms)

```
Constraint:     2 × BlockInterval(500ms) > 2 × RTT(240ms) + T_import(30ms) ✓
T_fill         = 250ms - T_constant(~30ms) = 220ms per block
CPU Utilization = 220 / 250 = 88%
```

The vote propagation cost is amortized across 2 block intervals (500ms total), easily satisfying the constraint. Each block interval is almost entirely devoted to transaction packing, with only constant overhead subtracted.

**Comparison:**

| Metric | N=1 (450ms) | N=2 (250ms) | Improvement |
| ------ | ----------- | ----------- | ----------- |
| CPU Utilization | 46.7% | 88% | +88.4% |
| T_fill per block | ~210ms | ~220ms | +4.8% |
| Blocks per second | 2.22 | 4.0 | +80% |
| Aggregate T_fill per second | ~467ms | ~880ms | +88.4% |

### 5.3 Impact on Builder Path

This optimization significantly improves validator packing capacity. However, the Builder path (MEV builder-proposer separation) is affected differently: shorter block intervals increase the proportion of constant overhead per block for builders, resulting in a moderate reduction in per-block builder capacity. The increased block frequency partially compensates in aggregate throughput.

## 6. Reward and Slash

### 6.1 Reward Adjustment

Under [BEP-126](./BEP126.md) and [BEP-319](./BEP-319.md), validators are rewarded for votes that are wrapped into attestations. With Vote Interval, voting occurs every N blocks instead of every block, reducing the number of reward events per epoch by a factor of N.

To maintain equivalent validator incentives:
- The **reward-per-attestation** should be scaled by a factor of N, so that total rewards per epoch remain unchanged.
- The reward distribution logic at epoch boundaries ([BEP-319](./BEP-319.md)) requires no structural changes — only the per-attestation weight is adjusted.

### 6.2 Slash Rules

The slash rules from [BEP-126](./BEP126.md) Section 4.4 apply to voting blocks:
- **Double vote**: A validator publishing two distinct votes for the same voting block height is slashable.
- **Surround vote**: A validator voting within the span of its other votes on the voting block sequence is slashable.
- The evidence submission deadline ([BEP-319](./BEP-319.md) Section 4.3) should be adjusted to account for the reduced number of voting blocks per unit time.

## 7. Security

This BEP preserves all safety guarantees of [BEP-126](./BEP126.md). The voting block subsequence forms a valid chain under the same consensus model:

- **Accountable Safety** (BEP-126 Theorem 1): The proof relies on super majority links between justified blocks. Since voting blocks maintain the same vote rules (no double vote, no surround vote) and the same ⅔+ threshold, the proof applies directly to the voting block subsequence. Two voting blocks on different forks cannot both be finalized unless ⅓+ validators violate vote rules.

- **Plausible Liveness** (BEP-126 Theorem 2): As long as ⅔+ validators are honest and follow the protocol, new voting blocks can always be justified and finalized without any validator violating vote rules.

- **Liveness of Block Production**: Non-voting blocks do not require Fast Finality participation. The chain can grow with ½+1 validators online, as in the original Parlia consensus. Fast Finality liveness (⅔+ online) is only required for the voting block subsequence.

- **Compatibility with existing FF optimizations**: [BEP-590](./BEP-590.md) (extended vote aggregation) and [BEP-648](./BEP-648.md) (in-memory voting pool) continue to function unchanged — they simply operate on the voting block subsequence rather than all blocks.

## 8. Backward Compatibility

This BEP is **not backward compatible**. It changes the Fast Finality voting and aggregation rules, requiring a hard fork with coordinated client upgrades across all validators and full nodes.

### 8.1 Block Finality

The fast finality mechanism now operates on voting blocks. With `N = 2` and `BlockInterval = 250ms`:

* **With fast finality working correctly**: Transaction finality requires 2 voting-block intervals, i.e., `2 × N × BlockInterval = 2 × 2 × 250ms = 1000ms`.
* **If fast finality fails** (natural finality with TurnLength=16, ValidatorSize=21):
  * \> 1/2 validator confirmations: 176 blocks (11×16) ≈ 44s
  * \> 2/3 validator confirmations: 240 blocks (15×16) ≈ 60s
