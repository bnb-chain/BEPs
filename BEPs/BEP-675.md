## 1. Summary

On BSC, the overwhelming majority of blocks are produced from builder bids (the MEV path) rather than purely local validator assembly. Under the existing PBS-style flow, the validator **simulates the winning bid** (re-executes its transactions) before sealing — duplicating execution the builder already performed. This BEP eliminates that validator-side simulation, shortens the builder pipeline critical path, and raises the achievable gas limit and overall throughput ceiling.

## 2. Abstract

Under the existing MEV mechanism, the critical path within a single block interval is:

**Import parent block → Builder constructs bid → Validator simulates bid → Block produced**

Validator-side bid simulation repeats execution the builder already performed, compressing the effective block-building window and limiting throughput. **Core idea:** delegate block construction more fully to the builder; the validator completes system-transaction signatures and sealing **without re-executing** user transactions on the assembly path. Builders submit **BidBlocks** with full execution results (stateRoot, receiptHash, etc.); the validator assembles and seals accordingly.

| Dimension | Current (`SendBid`) | Proposed (`SendBidBlock`) |
|-----------|---------------------|---------------------------|
| **Builder output** | Bid as a transaction list (plus declared reward); validator must replay to obtain execution results | **BidBlock**: signed user txs, `BidHeader` (execution results), unsigned system txs |
| **Validator work before seal** | **Bid simulation:** re-execute all transactions | Sign system txs → compute transaction root → assemble header → seal (**no EVM** on this path) |
| **EVM on the assembly critical path** | Import + builder work + **bid simulation** (three execution passes) | Import + builder work only (**no bid simulation**) |

## 3. Motivation

The primary goal of this proposal is to raise the practical gas limit on BSC. Under the legacy path, validator bid simulation consumes a significant share of each block interval, and that share grows proportionally with `gasLimit`—the higher the limit, the longer simulation takes, and the less time remains for builders to compete. By removing bid simulation from the validator's critical path, the time previously spent on redundant re-execution is reclaimed, making room for a substantially higher gas limit without compressing the bidding window.

## 4. Specification

### 4.1 Builder

A new RPC **`SendBidBlock`** is introduced alongside the legacy `SendBid`. Under `SendBid`, the validator re-executes the bid before sealing; under `SendBidBlock`, the validator **skips simulation** and uses the builder-supplied execution results directly (subject to post-seal verification described in [Section 6](#6-security)). Both paths coexist for backward compatibility.

The builder constructs and submits a BidBlock through the following steps:

1. **Execute user transactions** — run the selected user transactions (MEV bundles, mempool transactions, etc.) against the parent state.
2. **Execute system transactions in bid-mining mode** — run system transactions (fee distribution, slashing, reward distribution, validator set updates, etc.) through the EVM normally, but leave them **unsigned** (the builder does not hold the validator's private key). Signing does not affect EVM state changes, so the state root and receipts root are identical whether the system transactions are signed or not.
3. **Compute execution results** — derive the state root, receipts root, logs bloom, and gas used from the fully executed state.
4. **Declare `GasFee`** — set the expected net revenue for the validator, used for bid ranking.
5. **Assemble and sign `BidBlockArgs`** — pack user transactions, unsigned system transactions, `BidHeader` (execution results), declared `GasFee`, and optional blob sidecars into `BidBlockArgs`, then sign.
6. **Submit via `SendBidBlock`** — send the signed BidBlock through MEV-Sentry to the validator. Builders may submit improved versions until the deadline; a higher `GasFee` supersedes the prior best.

**`BidBlockArgs` data structure:**

```go
type BidBlockArgs struct {
    Signature   hexutil.Bytes   `json:"signature"`
    BlockNumber uint64          `json:"blockNumber"`
    ParentHash  common.Hash     `json:"parentHash"`
    GasFee      *big.Int        `json:"gasFee"`
    Txs         []hexutil.Bytes `json:"txs"`
    Header      *BidHeader      `json:"header"`
    SystemTxs   []hexutil.Bytes `json:"systemTxs"`
    Sidecars    []*BlobSidecar  `json:"sidecars,omitempty"`
}

type BidHeader struct {
    GasUsed     uint64      `json:"gasUsed"`
    Root        common.Hash `json:"stateRoot"`
    ReceiptHash common.Hash `json:"receiptsRoot"`
    LogsBloom   Bloom       `json:"logsBloom"`
}
```

| Field | Description |
|-------|-------------|
| `Signature` | Builder's signature covering block number, parent hash, declared fee, and execution header |
| `BlockNumber` / `ParentHash` | Used by the validator to reject stale or forked bids |
| `GasFee` | Declared net revenue for the validator, used for bid ranking |
| `Txs` | Signed user transactions (RLP-encoded) |
| `Header` | Builder-computed execution results (state root, receipts root, bloom, gas used) |
| `SystemTxs` | Unsigned system transactions (e.g. fee distribution, slashing) |
| `Sidecars` | Blob sidecars, if any blob transactions are included |

User transactions and system transactions are in **separate fields** because sender recovery (signature-based) is only possible on signed transactions. `BidHeader` carries only execution-result fields; consensus fields (block number, parent hash, timestamp, coinbase, difficulty, extra data, gas limit, base fee) are produced by the validator locally.

### 4.2 MEV-Sentry

Previously, MEV-Sentry participated in bid processing (including generating a dedicated payment transaction for the legacy `SendBid` path). On the `SendBidBlock` path, Sentry is reduced to a **pure bid proxy**: it forwards BidBlock traffic from builders to validators without additional processing. Its primary remaining role is to **conceal the validator's real network address** and protect validator infrastructure. Legacy `SendBid` handling remains for backward compatibility.

### 4.3 Validator

Upon receiving BidBlocks, the validator compares their declared `GasFee` and tracks the current best. At the assembly deadline (a short reserved interval before the block timestamp), the validator selects the highest-fee BidBlock and assembles the final block:

1. **Sign system transactions** — use the validator's private key to sign each unsigned system transaction from the BidBlock.
2. **Build the block header** — start from the validator's locally prepared consensus fields (block number, parent hash, timestamp, coinbase, extra data including the validator set snapshot at epoch boundaries, etc.), and fill in the execution-result fields (gas used, state root, receipts root, bloom) from the builder's `BidHeader`.
3. **Compute the transaction root** — calculate the Merkle root over the complete transaction list (user transactions + now-signed system transactions).
4. **Assemble the block** — combine the header and the full transaction list into a block.
5. **Collect vote attestation and seal** — aggregate Fast Finality vote attestations into the block header's extra data, then append the validator's own seal signature.

After sealing and broadcasting, the validator **imports the block locally** (re-executes all transactions and verifies the state root and receipts root). If import succeeds, the validator also compares the declared `GasFee` against the actual fee implied by execution. Mismatch or failure triggers builder penalties (see [Section 6](#6-security)).

### 4.4 Economic Model

**Payment transaction removal.** Under the legacy `SendBid` path, a dedicated on-chain payment transaction was used to transfer fees from the builder to the validator during simulation. With simulation removed on the `SendBidBlock` path, this payment transaction is **deprecated accordingly**.

Validator revenue instead comes solely from transaction gas fees within the block, distributed through the existing fee-distribution system transaction. `GasFee` in `BidBlockArgs` is a declared value used only for bid ranking; the validator's realized revenue is determined by on-chain execution.

With bid simulation removed, builders gain more time to optimize transaction ordering and value extraction without needing a separate payment mechanism.

## 5. Backward Compatibility

No hard fork is required: the change is at the **client implementation** layer and does not alter consensus rules. Externally observable on-chain block validity and execution semantics stay the same; only how the validator **assembles** the block before sealing changes.

Deployment is coordinated upgrades of builder, MEV-Sentry, and validator software to use `SendBidBlock`. The legacy `SendBid` path remains available; validators may fall back to simulation-based bids or local block building. A **gradual migration** (reference implementation first, then broader builder rollout, later deprecation of the legacy simulate-bid path) is recommended for live networks.

## 6. Security

Without simulation, the validator cannot fully verify block validity **before** sealing. Instead, correctness is enforced **after** broadcast: the validator imports the sealed block locally, re-executing all transactions and checking the state root and receipts root. Two categories of builder misbehavior are addressed:

| Scenario | Block validity | How detected | Impact |
|----------|---------------|--------------|--------|
| Builder submits incorrect execution results (e.g. wrong state root) | Invalid — other nodes also reject | Local import fails (state root mismatch) | Validator misses the block; builder is jailed |
| Builder declares inflated `GasFee` but provides correct execution results | Valid — block is accepted network-wide | Local import succeeds, but actual fee from execution is lower than declared | Builder is jailed for dishonest bidding |

Both cases are handled by a **local jail mechanism** on each validator. The jail mechanism is a **validator-local, off-chain** reputation system. Each validator independently maintains a jail list of builders, tracking offense counts and ban expiry. The mechanism operates as follows:

1. **Detection** — after sealing and broadcasting, the validator imports the block locally. If import fails (state root or receipts root mismatch) or the actual fee implied by execution deviates significantly from the declared `GasFee`, the builder is flagged.
2. **Short jail** — on first offense, the builder is barred from bidding for a configurable period (on the order of hours). During the jail period, the validator rejects all bids from this builder.
3. **Escalation** — each subsequent offense increments the builder's offense counter. Upon reaching a configurable threshold, the builder receives a longer ban (on the order of days) and the validator operator is notified for manual review.
4. **Permanent blacklist** — operators may choose to permanently blacklist a builder based on its offense history and severity.
