## 1. Summary

[BEP-221](./BEP221.md) introduced the CometBFT light block validation precompile at address `0x67` for cross-chain verification between BSC and CometBFT-compatible blockchains. The precompile decodes validator sets from both the trusted consensus state and the incoming light block, but does not enforce that validators within these sets are unique.

This BEP extends the `0x67` precompile with a Pasteur-activated variant that rejects any input containing duplicate validators — identified by validator address, consensus public key, BLS public key, or relayer address. This closes a gap between the uniqueness assumptions made by bridge verification logic and the actual guarantees enforced at the precompile boundary.

## 2. Motivation

The BSC native bridge relies on the `0x67` precompile to verify cross-chain light blocks. The bridge verification logic implicitly assumes that each validator in a given set is unique. However, the precompile itself does not enforce this invariant. A maliciously crafted input containing duplicate validators could exploit this gap in the following ways:

1. **Voting power inflation.** A duplicated validator effectively counts its voting power multiple times during signature verification. This could allow a minority validator set to appear as a supermajority, enabling forged cross-chain messages to pass verification.

2. **Signing threshold bypass.** During light client verification, the commit's signatures are matched against validators by their index position in the set. If the same public key occupies multiple indices, a single valid signature satisfies multiple slots, reducing the effective signing threshold required to pass verification.

3. **Bridge identity confusion.** Duplicate relayer addresses or BLS keys could cause ambiguous routing or signature attribution in the cross-chain relayer infrastructure.

Enforcing uniqueness at the precompile boundary eliminates this class of attacks regardless of how the calling contracts are implemented.

## 3. Specification

### 3.1 Background: Input Encoding

As defined in [BEP-221](./BEP221.md), the precompile input contains two parts:

```
|--consensus state length--|--consensus state--|--light block--|
|--32 bytes----------------|-------------------|---------------|
```

The **consensus state** encodes each validator in a fixed-width binary format:

```
|--validator pubkey--|--voting power--|--relayer address--|--relayer bls pubkey--|
|--32 bytes----------|--8 bytes-------|--20 bytes---------|--48 bytes------------|
```

The `relayer address` and `relayer bls pubkey` fields are bridge-specific extensions. When a validator has not registered its bridge identity on-chain, these fields are zero-filled. The validator's CometBFT address (20 bytes) is not stored explicitly — it is derived from the consensus public key during decoding via `tmhash(pubkey)[:20]`.

The **light block** uses CometBFT's native protobuf encoding (`LightBlock.Marshal`), which includes the validator address as an explicit field alongside the public key and other validator properties.

This BEP adds uniqueness validation to the validator sets from both parts of the input.

### 3.2 Uniqueness Validation Rules

When the Pasteur hardfork is active, the following four fields are checked for uniqueness across every validator within each validator set:

| Field | Width | Source | Uniqueness Requirement |
|-------|-------|--------|----------------------|
| Validator address | 20 bytes | Derived from pubkey (consensus state) or explicit (light block) | Strict — must be unique |
| Consensus public key (Ed25519) | 32 bytes | Explicit in both encodings | Strict — must be unique |
| BLS public key | 48 bytes | Explicit in both encodings | Unique among non-zero values (see §3.3) |
| Relayer address | 20 bytes | Explicit in both encodings | Unique among non-zero values (see §3.3) |

Since the validator address is derived from the consensus public key via a hash function, pubkey uniqueness implies address uniqueness in practice. The address check serves as a defensive redundancy layer.

Uniqueness is checked **independently** for the trusted consensus state validator set and the incoming light block validator set. Cross-set duplicates (a validator appearing in both sets) are not flagged, as this is expected during normal validator set transitions.

### 3.3 Handling of Unset Fields

BLS public keys and relayer addresses may remain unset for validators that have not registered their bridge identity. In the fixed-width encoding, an unset field is represented as all-zero bytes (48 zero bytes for BLS keys, 20 zero bytes for relayer addresses).

The uniqueness check applies the following rule:
- If a field value is **all-zero bytes** or **empty** (zero length), it is treated as **unset** and **excluded** from duplicate detection.
- Multiple validators may have unset BLS keys or relayer addresses without triggering an error.
- If any validator has a **non-zero** BLS key or relayer address that matches another validator's non-zero value in the same field, the precompile rejects the input.

In pseudocode, the skip condition is:

```
is_unset(value) = (len(value) == 0) OR (all bytes in value are 0x00)
```

### 3.4 Validation Points

The uniqueness check is applied at two points during precompile execution:

1. **Consensus state decoding** — after the trusted consensus state is successfully parsed from the fixed-width binary encoding (see §3.1), the uniqueness check is applied to its validator set. If duplicates are found, decoding fails.

2. **Light block decoding** — after the incoming light block is successfully parsed from the CometBFT protobuf encoding, the uniqueness check is applied to its validator set as well.

Both checks occur before any light client verification logic (signature verification, validator set hash comparison, etc.) runs, ensuring that malformed inputs are rejected as early as possible.

### 3.5 Error Behavior

When a duplicate is detected, the precompile returns an error with the following format:

```
duplicate validator <field> #<index1> and #<index2>: <hex_value>
```

For example:
```
duplicate validator address #0 and #3: 48D95FA953B2A3B6DBCEA298B7C1AE1D7B508CFB
```

The precompile execution fails and all provided gas is consumed, consistent with existing precompile error semantics on BSC.

### 3.6 Precompile Activation

The Pasteur hardfork introduces a new precompile mapping table `PrecompiledContractsPasteur`, cloned from `PrecompiledContractsOsaka`, with the `0x67` entry replaced by a new variant `cometBFTLightBlockValidatePasteur`. When `rules.IsPasteur` is true, the EVM selects this mapping table, and all calls to `0x67` are routed to the Pasteur variant.

The Pasteur variant inherits all existing validation behavior from the Hertz variant (i.e., `isHertz=true`) and additionally enables the uniqueness check (`requireUniqueValidators=true`). The three variant call signatures are:

| Variant | `isHertz` | `requireUniqueValidators` |
|---------|-----------|--------------------------|
| Original (pre-Hertz) | `false` | `false` |
| Hertz / Osaka | `true` | `false` |
| **Pasteur** | `true` | `true` |

Pre-Pasteur variants are unmodified and continue to accept duplicate validators.

## 4. Rationale

**Why enforce at the precompile level instead of in the calling contracts?**

The `0x67` precompile is a shared primitive used by the cross-chain light client smart contract. Relying on callers to sanitize validator sets is fragile: any new or upgraded contract that omits the check re-opens the vulnerability. Enforcing uniqueness in the precompile provides a single, authoritative validation point that all callers inherit automatically.

**Why not also modify the `0x66` (BLS signature verification) precompile?**

The `0x66` precompile is a generic BLS cryptographic primitive — it verifies an aggregated BLS signature against a set of public keys. Requiring unique public keys at this level would be overly restrictive, as there are legitimate contexts where the same key may appear in different roles. The bridge's signer uniqueness is already guaranteed by the validator set uniqueness enforced in `0x67`.

**Why skip all-zero optional fields?**

BLS keys and relayer addresses are set via an on-chain registration process that may lag behind validator set changes. During transition periods, multiple validators may legitimately have zero-filled bridge fields. Treating all-zero values as "unset" reflects the on-chain reality and avoids rejecting valid validator sets.

**Why check each validator set independently rather than merging them?**

The trusted consensus state and the incoming light block may share validators (e.g., when only a subset of the validator set rotates). Cross-set overlap is expected and correct. Only intra-set duplicates indicate malformed input.

## 5. Backward Compatibility

This is a **hard fork change** activated at the Pasteur upgrade. After activation, the precompile at `0x67` will reject inputs that it previously accepted — specifically, inputs containing duplicate validators within a single validator set.

Pre-Pasteur variants of the precompile are unmodified. Calls to `0x67` before Pasteur activation continue to behave identically to the current implementation.

In practice, this change does not affect legitimate cross-chain operations. Validator sets produced by honest CometBFT validators never contain duplicates. Only maliciously crafted inputs are rejected.

## 6. Security Considerations

- **Attack surface reduction.** This BEP closes a gap where the precompile's implicit assumption (unique validators) was not enforced, eliminating a class of potential bridge verification exploits.

- **Minimal code change.** The validation logic is purely additive — it checks a property of the decoded input before passing it to existing verification logic. The existing light client verification code is not modified.

- **Gas cost unchanged.** The gas cost of the `0x67` precompile remains identical to the pre-Pasteur version. The uniqueness check is O(n) in the number of validators, using hash maps for each field. For typical BSC validator set sizes (~40 validators), the overhead is negligible and no additional gas is charged.

- **Residual risk at `0x66`.** The BLS signature verification precompile at `0x66` does not enforce pubkey uniqueness. Callers that use `0x66` independently (outside the `0x67` light block flow) must ensure signer uniqueness themselves.
