## 1. Summary

[BEP-221](./BEP221.md) introduced the CometBFT light block validation precompile at address `0x67`. This precompile decodes validator sets from the trusted consensus state and the incoming light block, but does not enforce validator uniqueness.

This BEP adds a Pasteur-activated variant of `0x67` that rejects inputs containing duplicate validators, closing a gap between the uniqueness assumptions in bridge verification logic and the guarantees enforced at the precompile boundary.

## 2. Motivation

The bridge verification logic assumes each validator in a set is unique. Without enforcement at the precompile level, a crafted input with duplicate validators could:

- **Inflate voting power** — a duplicated validator counts multiple times, allowing a minority to appear as a supermajority.
- **Bypass signing thresholds** — the same public key at multiple indices lets one signature satisfy multiple slots in the commit.
- **Confuse bridge identity** — duplicate relayer addresses or BLS keys cause ambiguous attribution in the relayer layer.

## 3. Specification

### 3.1 Uniqueness Rules

The following fields are checked for uniqueness across every validator within each set:

| Field | Requirement |
|-------|-------------|
| Validator address (20 bytes) | Must be unique |
| Consensus public key (32 bytes) | Must be unique |
| BLS public key (48 bytes) | Must be unique among non-zero values |
| Relayer address (20 bytes) | Must be unique among non-zero values |

In the consensus state encoding ([BEP-221](./BEP221.md)), validator address is not stored explicitly — it is derived from the pubkey via `tmhash(pubkey)[:20]`. In the light block's protobuf encoding, it is an explicit field. The address check is a defensive redundancy since pubkey uniqueness already implies address uniqueness.

### 3.2 Unset Fields

BLS keys and relayer addresses may be unset (all-zero bytes) for validators that have not registered bridge identity on-chain. The uniqueness check skips any field value that is empty or all-zero:

```
is_unset(value) = (len(value) == 0) OR (all bytes are 0x00)
```

Multiple validators may have unset BLS keys or relayer addresses without triggering an error.

### 3.3 Validation Scope

The uniqueness check is applied to:

1. The validator set in the **trusted consensus state** (fixed-width binary encoding)
2. The validator set in the **incoming light block** (CometBFT protobuf encoding)

Both checks run before any light client verification logic. Cross-set duplicates are not flagged, as validators may appear in both sets during normal rotation.

When a duplicate is detected, the precompile returns an error:

```
duplicate validator <field> #<index1> and #<index2>: <hex_value>
```

### 3.4 Activation

The Pasteur hardfork introduces `PrecompiledContractsPasteur` (cloned from `PrecompiledContractsOsaka`), replacing the `0x67` entry with `cometBFTLightBlockValidatePasteur`. This variant inherits all Hertz validation behavior and additionally enforces uniqueness:

| Variant | `isHertz` | `requireUniqueValidators` |
|---------|-----------|--------------------------|
| Original (pre-Hertz) | `false` | `false` |
| Hertz / Osaka | `true` | `false` |
| **Pasteur** | `true` | `true` |

The `0x66` (BLS signature verification) precompile is not modified — it is a generic BLS primitive where pubkey uniqueness is the caller's responsibility.

## 4. Rationale

**Why at the precompile level?** Relying on callers to deduplicate is fragile — any new contract that omits the check re-opens the vulnerability. The precompile is the single authoritative validation point.

**Why skip zero-valued fields?** BLS keys and relayer addresses are set via on-chain registration that may lag behind validator set changes. Treating all-zero as "unset" avoids rejecting legitimate validator sets.

**Why check sets independently?** The trusted state and light block may share validators during rotation. Only intra-set duplicates indicate malformed input.

## 5. Backward Compatibility

This is a hard fork change. After Pasteur activation, `0x67` rejects inputs with duplicate validators that it previously accepted. Pre-Pasteur behavior is unchanged.

In practice, legitimate validator sets never contain duplicates. Only maliciously crafted inputs are affected.

## 6. Security Considerations

- **Gas cost unchanged.** The O(n) uniqueness check over ~40 validators is negligible. No gas adjustment is needed.
- **Residual risk at `0x66`.** Callers using the BLS precompile independently must enforce signer uniqueness themselves.
