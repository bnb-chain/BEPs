## 1. Summary

This BEP introduces a pre-confirmation mechanism using Relays and Sidecars to ensure Prop AMM transactions execute at top of the blocks.

## 2. Motivation

The goal is to address the insufficient support for Prop AMM on BSC, This proposal introduces a new mechanism to ensure that Prop AMM quote update transactions are executed at the top of a block, while maintaining extensibility for future pre-confirmation requirements.

The benefits for launching this mechanism are as follows:

- **Optimized Swap Execution**: Competition among multiple Prop AMMs and traditional AMMs allows traders to secure better execution prices whenever Prop AMM quotes outperform traditional liquidity models..
- **Increased Validator Revenue**: New quote update transactions generate additional fee revenue for validators, boosting the network's economic sustainability.

## 3. Background

### 3.1 Design Objectives

- **Objective 1**: Provide deterministic priority ordering for Prop AMM quote update  transactions, aiming for a stable confirmation window per block.
- **Objective 2**: Maintain full compatibility with the existing BSC network architecture.

### 3.2 Definitions

The system consists of five key roles:

1. **User**: The backend service of the party initiating Prop AMM quote update transactions.
2. **Preconf Relay**: The system's entry gateway. It performs security checks and authentication, then routes requests to the corresponding Validator's Preconf Sidecar based on the global distribution of BSC Validators .
3. **Preconf Sidecar**: A core coordination middleware deployed alongside each Validator. It handles business logic, coordinates constraint creation, distributes Inclusion Lists to Builders, validates bids, and drives final block construction.
4. **Validator**: Responsible for final block simulation and construction.
5. **Builder**: Receives the Inclusion List (transactions must be included at top of the block) from the Sidecar and packages transactions according to the specified requirements.

## 4. WorkFlow

![multiple phases](./assets/bep-668/4-WorkFlow.png)

### 4.1 Request Submission & Routing

- **Submit preconf request**: The User sends a pre-confirmation request (containing the quote update  transaction) to the Relay.
- **Authorize request**: Upon receipt, the Relay performs internal authentication, verifying user identity, signatures, and request validity.
- **Route request**: After successful authorization, the Relay forwards the request to the appropriate Preconf Sidecar.

### 4.2 Constraint Building & Signing

- **Build constraints**: The Sidecar notifies the Validator to build constraints based on the current request, ensuring the Prop AMM transaction is included and prioritized ahead of other transaction types.
- **Signed inclusion list**: An Inclusion List is generated and signed, containing the transactions and the Sidecar's signature endorsement.

### 4.3 Building, Bidding & Block Proposing

- **Send inclusion list to connected builders**: The Sidecar broadcasts the signed Inclusion List to all connected Builders, requiring them to adhere to the list and its ordering constraints.
- **Submit bid with inclusion list**: Builders package transactions accordingly, construct the block, and submit a bid to the Sidecar.
- **Validate bid**: The Sidecar validates the received bid, checking list integrity, bid validity, and ordering constraints.
- **Simulate bid and build block**: After validation, the Sidecar passes the bid information to the Validator. The Validator performs a final simulation, completes the block construction and proposes the block.

## 5. Security Design

Three types of anomalies are addressed to prevent potential losses for Prop AMM:

1. **Slashing**: We recommend the Prop AMM contracts should integrate a state verification mechanism based on block height and difficulty. If a slashing event is detected, this mechanism acts as a circuit breaker, forcing a revert of all Prop AMM calls in the current block.
2. **Validator Proposing Non-MEV Blocks / Component Downtime**:
   - **Option 1: Heartbeat/Anchor Transaction**: We recommend the Prop AMM providers  should  submit an anchor transaction at every block height as an activation signal. If the block lacks this signal, all related Prop AMM transactions automatically fail.
   - **Option 2: Mandatory Local Ordering**: As the second option, we propose that the BNB chain team can  implement strict ordering constraints in the validator's local block-building codes to prioritize pre-confirmed transactions at the top of blocks.
